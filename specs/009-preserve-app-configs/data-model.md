# Phase 1 Data Model — Preserve Operator-Edited Per-App Routing Files

**Date**: 2026-05-01

This feature introduces no new persisted entities. It changes the *lifecycle* of an existing on-disk artifact (the per-app routing file) and adds one in-memory field to one struct. This document is therefore short.

## Per-App Routing File Lifecycle

The per-app routing file (`<routingDir>/dynamic/<team>-<service>.yml`) has the following states under this feature. States are determined by what is on disk, not by any Shrine-managed record.

```text
                               operator rm
                          ┌────────────────────────┐
                          ▼                        │
                       Absent                  Operator-Owned
                          │                        ▲
                          │                        │
              first deploy│                        │repeat deploys (no-op)
                          ▼                        │
                      Generated  ─────────────────►│
                          │
                          │ teardown of app
                          ▼
                       Orphan
                          │
                          │ operator rm
                          ▼
                       Absent
```

| State | What's on disk | Shrine's policy |
|---|---|---|
| **Absent** | nothing at the path | Next deploy of an app with this name will write a fresh file (`gateway.route.generated`). Teardown of an app with this name is a silent no-op. |
| **Generated** | bytes that match what `WriteRoute` would produce from the current manifest | Indistinguishable from "Operator-Owned" by Shrine — Shrine never reads the file back. From the moment a deploy writes the file, Shrine treats it as operator-owned on every subsequent operation. |
| **Operator-Owned** | bytes the operator hand-edited (or untouched-but-stale generated bytes) | Preserve on every deploy (`gateway.route.preserved`). Operator must `rm` to force regeneration. |
| **Orphan** | bytes that exist after the backing app has been torn down | Warn on every teardown (`gateway.route.orphan`); never auto-delete. Operator must `rm` to fully tear down the route. |

**Important**: Shrine cannot distinguish "Generated" from "Operator-Owned" — by design (FR-008: do not inspect content). The state diagram shows them as a single conceptual state from Shrine's point of view. The distinction matters only to the operator, who knows whether they edited the file.

The "Orphan" state is an **operator-action state**: Shrine flags it but does not transition out of it. Only an operator `rm` returns the path to "Absent".

## Validation Rules

There are no field-level validation rules for this feature. The single rule is structural: the per-app file path is always `<routingDir>/dynamic/<team>-<service>.yml`, where `<team>-<service>` is computed by `routeFileName(team, service)` (`internal/plugins/gateway/traefik/config_gen.go:103`). This name is already deterministic and unique per `(team, service)` tuple; this feature does not change that.

## In-Memory Struct Extension

One field is added to one struct. No new types.

**`internal/plugins/gateway/traefik/routing.go`**:

```go
type RoutingBackend struct {
    routingDir string
    observer   engine.Observer  // NEW — set by Plugin.RoutingBackend() from p.observer
}
```

The `observer` field is set once at construction (in `Plugin.RoutingBackend()`) and is read-only for the lifetime of the struct. There is no synchronization concern: each `Engine` has at most one `RoutingBackend` and the engine drives it serially via `ExecuteDeploy` / `ExecuteTeardown`.

The struct is constructed exactly twice in the codebase:

1. **Real backend**: `Plugin.RoutingBackend()` at `plugin.go:158-164` — passes `p.observer`.
2. **Test backend**: `routing_test.go:37-39` `newTestBackend()` — extended to pass a `&recordingObserver{}` (the existing test-double pattern from `config_gen_test.go:16-19`).

No other construction sites exist.

## State Transitions Reference Implementation

The four transitions Shrine controls map 1:1 to four code paths:

| Transition | Code path | Event |
|---|---|---|
| Absent → Generated | `WriteRoute` after `isPathPresent == false` and `os.WriteFile` succeeds | `gateway.route.generated` (Info) |
| Generated → Operator-Owned | (no transition; operator action, observable only via file-content drift on disk) | — |
| Operator-Owned → Operator-Owned (deploy is a no-op) | `WriteRoute` after `isPathPresent == true` | `gateway.route.preserved` (Info) |
| Operator-Owned/Generated → Orphan (teardown leaves file behind) | `RemoveRoute` after `isPathPresent == true` | `gateway.route.orphan` (Warning) |

The fifth path — stat error — is not a state transition; it's an unknown-state observation:

| Observation | Code path | Event |
|---|---|---|
| State unknown (Lstat returned non-`IsNotExist` error) | `WriteRoute` or `RemoveRoute` after `isPathPresent` returns an error | `gateway.route.stat_error` (Warning) |

When a stat error occurs, Shrine treats the file as present (does not write, does not delete) and continues the deploy. The state from Shrine's point of view is "unknown but not Absent"; the operator resolves it by inspecting the path.
