## Open Questions Resolved

The plan's Phase 0 listed six open questions; each is resolved below from the matrix above plus pre-survey reads.

1. **Does `routing_test.go` assert NO strip middleware for `stripPrefix: false`?** Yes — `routing_test.go:231-232` and `routing_test.go:288` both assert non-existence at the YAML-string level (`!strings.Contains(got, "middlewares:")` and `!strings.Contains(got, "strip-2:")`). The assertion shape is "no middleware emitted" not just "middleware list does not contain X." FR-003 evidence is unambiguous.
2. **Does `parser_test.go` cover nil/true/false unmarshal?** Nil is covered (`parser_test.go:106-108`, `parser_test.go:132-134`, `parser_test.go:142-144`); explicit `false` is covered (`parser_test.go:152-156`); explicit `true` is covered structurally by `engine_aliases_test.go:30-35` against the engine-side projection. The audit treats this as sufficient — the YAML library (`gopkg.in/yaml.v3`) is the source of truth for `*bool` unmarshal, and adding a parser-level explicit-true case would be redundant under YAGNI.
3. **Does `validate_test.go` confirm `stripPrefix: false` is valid?** Yes — `validate_test.go:306-313` ("valid alias with explicit stripPrefix false") asserts `wantErr: ""`.
4. **Does `traefik_plugin_test.go` produce a YAML-level integration assertion that `strip-2` is absent for `stripPrefix:false`?** Yes — `traefik_plugin_test.go:619-621` (multi-alias scenario, alias-2). Runs the real shrine binary as a subprocess and inspects the dynamic-config file on disk. Constitution V is satisfied for FR-003 by this assertion.
5. **Is there a third operator-facing doc home beyond `manifest-schema.md` and `AGENTS.md`?** No. `README.md` does not exist (`/root/projects/shrine/README.md` absent at audit time). `docs/` directory does not exist. The repo's documentation surface area is the schema contracts under `specs/*/contracts/` plus `AGENTS.md` and `CLAUDE.md`. The two-home choice from spec clarification Q2 covers the full operator-facing surface.
6. **What is the exact current shape of the per-alias log line?** `formatAliasesForLog` (`engine.go:324-335`) returns `strings.Join(entries, ",")` where each entry is either `r.Host` (no path prefix) or `r.Host+"+"+r.PathPrefix`, sorted ascending alphabetically. The marker design: append `" (no strip)"` (with leading space) to entries that have `r.PathPrefix != "" && !r.StripPrefix`. This keeps the comma separator unaffected and keeps the existing entry shape character-for-character for stripping aliases.

## Decisions

### Decision 1: Marker syntax — `" (no strip)"` suffix on the entry, not a separate field

- **Decision**: Append literal `" (no strip)"` (one leading space) to the per-entry string for aliases that match `r.PathPrefix != "" && !r.StripPrefix`. The comma-joined output remains the value of the existing `aliases` event field.
- **Rationale**: The existing event field is a single string, not a structured list. Adding a sibling field (e.g., `aliases_nostrip`) would change the event schema and break any current log-shape consumers; the spec's compatibility constraint (SC-004) and FR-009 prefer additive marker syntax. The leading space prevents the marker from being mistaken for part of the path; parentheses are visually distinct from `+` (the path-prefix separator) and from `,` (the entry separator).
- **Alternatives considered**:
  - *Sibling event field `aliases_nostrip`*: rejected — schema change, more code, no operator-visible benefit beyond what an inline marker provides.
  - *Different separator (e.g., `!nostrip`, `[nostrip]`, `*`)*: rejected — `(no strip)` is the most plain-English option and the spec's example marker. Plain English wins for an operator-facing log signal.
  - *Mark every alias with its strip state (`(strip)` and `(no strip)`)*: rejected — violates SC-004 / FR-009 (default-strip aliases must keep their existing log shape character-for-character).

### Decision 2: No HTTP-level integration test under this feature

- **Decision**: Per spec clarification Q4, do not add a new integration scenario that deploys a real Next.js (or any echo-path) backend, sends HTTP requests, and asserts the path arrives intact. The existing `tests/integration/traefik_plugin_test.go:602-621` YAML-level assertion is the integration-gate equivalent for the bug class.
- **Rationale**: The bug class is "wrong middleware composition." Once the dynamic-config YAML is correct (no strip middleware attached when `stripPrefix: false`), Traefik's runtime behavior follows by definition — Traefik does not have a separate code path that re-strips paths despite missing middleware. The full HTTP reproduction would prove that Traefik works as documented, which is upstream's responsibility, not Shrine's. Per memory, the integration suite is ~10 minutes per pass; adding scenarios with marginal proof value harms iteration speed.
- **Alternatives considered**:
  - *Full HTTP reproduction with real backend*: rejected per Q4. Would prove Traefik upstream behavior, not Shrine's correctness.
  - *Skip new tests entirely*: rejected — FR-010 is a Gap and needs unit-level coverage (`engine_aliases_test.go` extension). The audit step itself does not constitute new test coverage.

### Decision 3: Keep `formatAliasesForLog` as a single function (no extraction)

- **Decision**: Add the marker logic inline inside `formatAliasesForLog`. Do not extract `formatOneAliasEntry` or `joinAliasEntries` helpers.
- **Rationale**: The function will be ~10 lines. Extracting helpers when there is one call site violates Constitution Principle IV (YAGNI: abstractions justified by ≥3 concrete usages). The post-modification function is still readable; the conditional `if r.PathPrefix != "" && !r.StripPrefix { entry += " (no strip)" }` is self-documenting.
- **Alternatives considered**:
  - *Extract `formatOneAliasEntry(r AliasRoute) string` and `joinAliasEntries(entries []string) string`*: rejected — single-call-site extraction. Re-evaluate if a third call site appears later.
